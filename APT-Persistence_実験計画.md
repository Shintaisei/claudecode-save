# APT-Persistence 実験計画
> 作成日: 2026-04-15  
> 前提資料: `APT-Persistence_ホスト選定分析.md`, `APT-Persistence_研究方針まとめ.md`  
> 主対象ホスト: `C_Data/96`  
> 比較候補: `C_Data/42`, `C_Data/80`  
> 補助候補: `C_Data/63`, `C_Data/30`

---

## 1. 現在地

研究の第一段階は、**起点ログから正常行動の関連ログをたどり、正常行動として復元できるかを検証すること**である。

ここまでで、データセット選定とホスト選定は完了した。

| 項目 | 現状 |
|---|---|
| データセット | apt-persistence を採用 |
| 主対象 | `C_Data/96` |
| 比較候補 | `C_Data/42`, `C_Data/80` |
| 予備分析済み | `C_Data/30` |
| 分析対象ログ | `Sysmon.evtx`, `Security.evtx`, `TaskScheduler.evtx` |
| 起点候補 | Wazuh アラート、必要に応じて Hayabusa アラート |

主対象である `C_Data/96` は、正常ホストでありながら Wazuh アラートが 1,458 件あり、Sysmon/Security/TaskScheduler もそろっている。そのため、正常行動復元の実験対象として十分に使える。

---

## 2. Hayabusa は必要か

結論から言うと、**ホスト選定のためには Hayabusa はもう不要**である。

ただし、研究の本体である「起点アラートから正常行動を復元する」実験では、Hayabusa を使う意味はまだある。

### 2.1 Hayabusa が不要な部分

以下の目的であれば、Hayabusa は不要。

| 目的 | 理由 |
|---|---|
| 研究対象ホストを決める | 既に `C_Data/96` に決定済み |
| 正常ホストのログ量を比較する | EVTXサイズ、Sysmon期間、Wazuh件数で判断済み |
| 予備的な起点候補を探す | Wazuhアラートが既に存在する |
| 正常行動の大まかな傾向を見る | Sysmon/Wazuh/Description から確認可能 |

つまり、**「どのホストを使うか」や「正常ログにアラートがあるか」を確認するためだけなら Hayabusa を回す必要はない**。

### 2.2 Hayabusa を使う意味がある部分

一方で、以下の目的では Hayabusa を使う価値がある。

| 目的 | Hayabusaを使う意味 |
|---|---|
| 実運用寄りの起点アラートを作る | EVTXに対するルールベース検知として説明しやすい |
| 網屋MTGでの説明材料にする | 「Hayabusaで出たアラートから正常行動を復元した」と言える |
| Wazuh依存を避ける | データセット付属のWazuhではなく、こちらで再現可能な起点を作れる |
| HayabusaとWazuhの差を見る | 同じ正常ログに対して、どの検知が出るか比較できる |
| 将来の自動化に接続する | Hayabusa出力を入力にして関連ログ探索する流れを作れる |

したがって、Hayabusa は **ホスト選定ツールではなく、実験用の起点アラート生成ツール**として位置づけるのがよい。

## 3. 推奨方針

推奨は、**まず Wazuh アラートだけで正常行動復元の手順を作り、その後に Hayabusa で同じ手順を検証する**ことである。

理由は、Wazuhアラートは既に手元にあり、すぐ起点候補を選べるからである。最初からHayabusa待ちにすると、実験が止まる。

### 方針A: Wazuh先行

最初にやるべき方針。

| 項目 | 内容 |
|---|---|
| 起点 | `C_Data/96/Wazuh-Alerts/alerts.json` |
| 対象ログ | `Sysmon.evtx`, `Security.evtx`, `TaskScheduler.evtx` |
| 目的 | 起点アラートから正常行動を復元する手順を確立する |
| 利点 | 今すぐ始められる |
| 欠点 | Hayabusaを使った実験とは言えない |

### 方針B: Hayabusa追加

Wazuh先行で手順が固まった後にやる。

| 項目 | 内容 |
|---|---|
| 起点 | Hayabusaの検知結果 |
| 対象ログ | `C_Data/96` のEVTX |
| 目的 | 実運用に近い検知結果を起点に復元できるか確認する |
| 利点 | 研究の説明力が上がる |
| 欠点 | Hayabusa実行環境と出力整形が必要 |

### 方針C: Hayabusaなし

時間がない場合の現実的な選択肢。

| 項目 | 内容 |
|---|---|
| 起点 | Wazuhアラートのみ |
| 目的 | 正常行動復元の成立性を示す |
| 利点 | すぐに分析を進められる |
| 欠点 | 「Hayabusaにかける」という当初説明からは少し変わる |

今回の研究発表までを考えると、**方針Aを必須、方針Bを余裕があれば実施**がよい。

## 4. 実験の問い

実験で答えるべき問いは以下。

### RQ1: 正常ホストで出たアラートを起点に、関連ログをたどれるか

見るもの:

- 起点アラートの時刻
- 起点アラートのプロセス名
- Sysmonの同時刻周辺イベント
- 親子プロセス関係
- ユーザー、ログオン、タスク実行

成功条件:

- 起点アラートの前後に関連ログが見つかる
- 主体、行為、対象、連鎖を説明できる

### RQ2: 復元した行動を正常行動として説明できるか

見るもの:

- インストール済みソフトとの整合
- Windows標準動作との整合
- Wazuh/Hayabusaのルール説明との整合
- 時系列上の自然さ

成功条件:

- 「攻撃ではなく正常行動」と判断する根拠を文章化できる

### RQ3: 正常と断定できないケースを分離できるか

見るもの:

- 関連ログが不足しているケース
- 主体や親プロセスが不明なケース
- 正常ソフトと結びつかないケース

成功条件:

- 「正常」「判断不能」「追加調査必要」を区別できる

## 5. 実験手順

### Step 1: 起点アラート候補を抽出する

対象は `C_Data/96` の Wazuh アラート。

まず以下のカテゴリを優先する。

| 優先度 | カテゴリ | 理由 |
|---:|---|---|
| 1 | Discovery | 攻撃にも正常管理にも出やすく、復元の題材にしやすい |
| 2 | script / PowerShell | 正常な開発・管理作業と攻撃の境界になりやすい |
| 3 | TaskScheduler / taskschd.dll | Persistence類似として研究テーマに近い |
| 4 | privilege / logon | 認証・権限文脈と結びつけやすい |
| 5 | file / registry | ソフトウェア更新やインストール行動と結びつけやすい |

`C_Data/96` で確認済みの主な候補:

- `Failed attempt to perform a privileged operation.`
- `Discovery activity executed`
- `Process loaded taskschd.dll module. May be used to create delayed malware execution`
- `Scripting file created under Windows Temp or User folder`
- `Windows logon success.`

最初は、件数が多すぎる `Failed attempt to perform a privileged operation.` よりも、**Discovery / taskschd.dll / scripting file** を優先する。

### Step 2: 起点イベントを1件選ぶ

各カテゴリから代表イベントを1〜2件選ぶ。

選定条件:

- 時刻が明確
- プロセス名が明確
- Sysmonで同時刻周辺にイベントがある
- Description.yml のソフトウェア構成と結びつく可能性がある

最初の分析では、合計3件程度でよい。

### Step 3: 前後時間窓を切る

起点時刻を中心に、以下の時間窓で関連ログを探索する。

| 時間窓 | 用途 |
|---|---|
| 前後5分 | 直接関連イベントを探す |
| 前後30分 | 親プロセス、ログオン、タスク、通信を探す |
| 前後2時間 | ソフトウェア更新や環境構築など長い行動を確認する |

最初は前後30分で十分。足りなければ2時間に広げる。

### Step 4: Sysmonで行動連鎖を復元する

主に見るSysmonイベント:

| Event ID | 内容 | 使い方 |
|---:|---|---|
| 1 | Process Create | プロセス親子関係、コマンドライン |
| 3 | Network Connection | 外部通信、プロキシ通信 |
| 11 | File Create | スクリプト、インストーラ、一時ファイル |
| 12/13/14 | Registry | Run系、設定変更、ソフト更新 |
| 22 | DNS Query | アクセス先の推定 |

復元する要素:

- 主体: どのユーザー、どのプロセスか
- 行為: 何をしたか
- 対象: どのファイル、通信先、レジストリか
- 連鎖: 親プロセス、子プロセス、前後イベント
- 根拠: どのログで説明できるか

### Step 5: Security / TaskSchedulerで補強する

Securityで見るもの:

- ログオン成功
- 監査失敗
- 特権操作
- プロセス生成の補助情報

TaskSchedulerで見るもの:

- タスク起動
- タスク終了
- Windows標準タスク
- Edge/Defender/Update系タスク

Sysmonだけで説明できる場合もあるが、SecurityやTaskSchedulerがあると「正常行動としての説明」が強くなる。

### Step 6: 正常行動として分類する

復元結果は以下に分類する。

| 判定 | 条件 |
|---|---|
| 正常 | ソフトウェア構成、親子関係、時系列から自然に説明できる |
| 正常寄り | 説明はできるが、一部ログが不足している |
| 判断不能 | 主体、対象、連鎖のいずれかが欠ける |
| 要追加調査 | 正常ソフトと結びつかず、攻撃類似性が残る |

今回の研究では、無理に全部を正常と断定しない。**判断不能を分離できることも成果**として扱う。

## 7. 優先タスク

### 必須

1. `C_Data/96` の Wazuh アラートをカテゴリ別に集計する
2. Discovery / taskschd.dll / scripting file から起点候補を3件選ぶ
3. 各起点の前後30分の Sysmon/Security/TaskScheduler を抽出する
4. プロセス親子関係と時系列を整理する
5. 3件分の正常行動復元ケースを文章化する

### 余裕があれば

1. `C_Data/42` でも同じ手順を1件だけ試す
2. Hayabusaを `C_Data/96` にかける
3. Wazuh起点とHayabusa起点の違いを比較する
4. `C_Data/30` を予備分析例として発表資料に入れる

### 後回し

1. 全アラートの網羅分析
2. すべての正常ホストへの展開
3. 攻撃ホスト `I_Data` との比較
4. 自動判定モデルの実装

## 8. Hayabusaを使う場合の実験設計

Hayabusaを使う場合は、目的を以下に限定する。

> C_Data/96 の正常EVTXに対して Hayabusa が出すアラートを起点に、正常行動復元ができるかを確認する。

やること:

1. `C_Data/96/Evtx_Logs` を Hayabusa に入力する
2. CSVまたはJSONで結果を出力する
3. severity、rule title、timestamp、computer、channel、eventID を整理する
4. Wazuhと同じカテゴリにマッピングする
5. Wazuhで作った復元手順を適用する

Hayabusa実行後に見るべき点:

| 観点 | 内容 |
|---|---|
| Wazuhと同じ起点が出るか | 同じ正常行動を別検知器でも拾うか |
| Hayabusa特有の起点があるか | Sigma由来の検知で新しい題材が出るか |
| 復元しやすいか | ルール名とEVTXイベントが行動復元に使いやすいか |
| ノイズ量 | 起点候補として扱える件数か |

Hayabusaを使った場合の発表上の言い方:

> ホスト選定はログ期間・ログ種別・Wazuhアラート・ソフトウェア構成で完了した。Hayabusaは選定のためではなく、実運用に近い起点アラートを生成し、その起点から正常行動を復元できるかを確認するために使用する。

---

## 9. Hayabusaを使わない場合の実験設計

Hayabusaを使わない場合は、研究の説明を以下のように変える。

> apt-persistence に付属する Wazuh アラートを起点として、正常ホストにおけるアラート発生行動を復元する。

この場合のメリット:

- すぐに分析を始められる
- アラートとEVTXが同じデータセット内にそろっている
- C_Dataは正常ホストなので、出たアラートは偽陽性・正常行動候補として扱える

この場合の弱点:

- Hayabusa前提の説明からは外れる
- 網屋の異常検知システムとの接続が少し弱くなる
- Wazuh固有のルールに依存する

発表での言い方:

> 当初はHayabusaへの投入を想定していたが、apt-persistenceにはWazuhアラートが付属しており、正常ホストに対する起点アラートが既に得られている。そのため、第一段階ではWazuhアラートを起点として正常行動復元手順を作成し、Hayabusaは追加検証として位置づける。

## 11. 発表用の整理

発表では、以下の流れにすると自然。

1. 目的は、攻撃再現だけでなく、偽陽性を正常と判断する分析を自動化すること
2. その第一段階として、起点アラートから正常行動を復元する
3. データセットは EVTX と正常ログを持つ apt-persistence を採用した
4. 正常112台を段階的に評価し、C_Data/96 を主対象にした
5. ホスト選定には Hayabusa は不要になった
6. ただし、起点アラート生成・実運用寄り検証として Hayabusa を使う価値は残る
7. まずは付属Wazuhアラートで復元手順を作り、余裕があれば Hayabusa で再検証する

---

## 12. まとめ

この先の実験は、Hayabusaを回すこと自体が目的ではない。

重要なのは、**正常ホストに出たアラートを起点として、関連ログをたどり、正常行動として説明できるか**である。

そのため、次にやるべきことは、Hayabusa実行ではなく、まず `C_Data/96` の Wazuh アラートから起点候補を選び、3件程度の正常行動復元ケースを作ることである。

Hayabusaは、その手順ができた後に、同じ復元手順が別の検知器の出力にも適用できるかを確認する追加実験として扱う。
