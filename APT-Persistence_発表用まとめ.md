## はじめに：このドキュメントで何がわかるか

このドキュメントでは、研究で使用する **apt-persistence** というデータセットについて説明します。
「そもそもWindowsのログって何？」というところから始めて、このデータセットが研究にとってどう役立つのかまで順番に説明します。

## 第1章：前提知識 — Windowsのログとは何か

### 1-1. ログとは「コンピュータの行動記録」

コンピュータは何かが起きるたびに記録を残します。たとえば：

- 誰かがログインした → 記録
- あるアプリが起動した → 記録
- ファイルが作成・削除された → 記録
- インターネットに接続した → 記録

この記録のことを **ログ（Log）** と呼びます。

### 1-2. Windowsのログは「イベントログ」という仕組みで管理される

Windowsには OS 標準の記録システムがあり、これを **Windows イベントログ** と呼びます。
すべての記録は **`.evtx` という形式のファイル** に保存されます。

> **EVTX = Windows Event Log の拡張子。Windowsが標準で使うログファイルの形式。**

イベントログにはいくつかの種類があります：

| ログ名 | 何を記録するか | 主なイベントID（EID）の例 |
|--------|--------------|--------------------------|
| **Security.evtx** | 誰がいつログインしたか、どんなプロセスが起動したか、権限の使用 | EID 4624（ログイン成功）、EID 4688（プロセス起動）|
| **System.evtx** | OSレベルのイベント（起動・シャットダウン、サービスの開始・停止）| EID 7036（サービス状態変化）|
| **Application.evtx** | アプリケーションのエラーや動作イベント | アプリごとに異なる |
| **TaskScheduler.evtx** | スケジュールタスクの作成・実行・完了 | EID 200（タスク実行開始）|

> **イベントID（EID）とは？**
> Windowsが「何が起きたか」を数字で分類した番号。たとえばEID 4624は「ログイン成功」を必ず意味する。この番号があることで、膨大なログの中から「ログイン系のイベントだけ」を素早く絞り込める。

### 1-3. Sysmon：標準ログの弱点を補う「拡張監視ツール」

Windows標準のイベントログには弱点があります。
たとえばEID 4688（プロセス起動）は「何かのプロセスが起動した」とは記録しますが、**起動時にどんなコマンドが使われたか（CommandLine）は記録されない**場合があります。

それを補うために使われるのが **Sysmon（System Monitor）** です。

- Microsoftが無償で公開している追加監視ツール
- インストールすると、標準では取れない細かい情報まで記録するようになる
- **Sysmon.evtx** というファイルに保存される

Sysmonが記録するイベントの主な種類：

| Sysmon EID | イベント名 | 何を記録するか |
|-----------|-----------|--------------|
| **1** | ProcessCreate | プロセス起動。**実行コマンド・起動元プロセス・ユーザー名**まで記録 |
| **3** | NetworkConnect | ネットワーク接続。**どのプロセスがどのIPに接続したか** |
| **7** | ImageLoad | DLL（プログラム部品）のロード |
| **10** | ProcessAccess | あるプロセスが別のプロセスにアクセスした |
| **11** | FileCreate | ファイルの作成 |
| **12/13** | RegistryCreate/Set | レジストリの作成・変更 |
| **22** | DnsQuery | DNS名前解決（どのドメインを調べたか）|

> **なぜSysmon EID 1が重要か？**
> `ProcessGuid`（プロセスの一意ID）と `ParentProcessGuid`（親プロセスのID）が記録される。これにより「AというプロセスがBを起動し、BがCを起動した」という**連鎖（プロセスツリー）**を完全に再現できる。

### 1-4. MITRE ATT&CK：攻撃手法を整理した「攻撃の辞書」

**MITRE ATT&CK（マイターアタック）** とは、世界中で観測された実際の攻撃手法を体系的にまとめたデータベースです。

- 攻撃の各段階（侵入・権限昇格・横移動・永続化 など）を **タクティクス（戦術）** として分類
- 各段階でどんな具体的な方法が使われるかを **テクニック（技術）** として番号付きで管理

例：

| テクニックID | 名前 | 意味 |
|------------|------|------|
| **T1053.005** | Scheduled Task | スケジュールタスクを使って悪意あるプログラムを自動実行させる |
| **T1547.001** | Registry Run Keys | レジストリにプログラムを登録してOS起動時に自動実行させる |
| **T1112** | Modify Registry | レジストリを書き換えて設定を改ざんする |

> このデータセットの攻撃はすべてMITRE ATT&CKのテクニックIDと対応付けられている。

### 1-5. Hayabusa：Windowsログを高速に分析するツール

**Hayabusa** は、EVTX形式のWindowsログを入力として受け取り、「このログはどんな攻撃の可能性があるか」を自動で判定するオープンソースツールです。

- 内部的には **Sigmaルール**（攻撃パターンを記述したルール集）を使って判定する
- 出力は「アラート（警告）」として出てくる
- **EVTX形式でないと動かない**という制約がある

## 第2章：apt-persistence データセットとは何か

### 2-1. 一言で言うと

> **「Windows VM（仮想マシン）138台分のイベントログを、正常環境（112台）と攻撃が注入された環境（26台）に分けて収録した、EVTX形式の公開データセット」**

| 項目 | 内容 |
|------|------|
| 配布元 | cylab.be（ベルギー王立軍事アカデミー Cyber Defence Lab）|
| 出典論文 | Rahal, Riahi, Debatty — ICIN 2025（DOI: 10.1109/ICIN64016.2025.10943025）|
| 攻撃のカバー範囲 | MITRE ATT&CK Persistence（TA0003）：19テクニック・67サブテクニック |

### 2-2. データの全体構造

```
apt-persistence/Datasets/
│
├── C_Data/        ← 【正常】112台のVMログ（攻撃なし）
│   ├── 1/
│   ├── 2/
│   ├── ...
│   └── 112/
│
└── I_Data/        ← 【攻撃注入済み】26台のVMログ
    ├── 1/
    ├── ...
    └── 26/
```

**C_Data（Clean Data）** = きれいな環境。正常な使い方だけのログ。
**I_Data（Infected Data）** = 攻撃が仕掛けられた環境。正常ログの中に攻撃の痕跡が混ざっている。

### 2-3. 各VMフォルダの中身（共通構造）

```
{番号}/
├── Evtx_Logs/
│   ├── Security.evtx          ← 認証・プロセス・権限（Windowsセキュリティログ）
│   ├── Sysmon.evtx            ← プロセス連鎖・ネットワーク・ファイル・レジストリ
│   ├── TaskScheduler.evtx     ← スケジュールタスクの実行ログ
│   ├── Application.evtx       ← アプリのエラー・動作ログ
│   └── System.evtx            ← OSシステムイベント
│
├── Wazuh-Alerts/
│   └── alerts.json            ← WazuhというSIEMが自動生成したアラート一覧
│
└── Description/
    ├── Description.yml        ← そのVMのOS・インストールソフト情報
    └── GT.yml                 ← 【I_Dataのみ】実施した攻撃のMITRE IDとコマンド
```

> **SIEMとは？**
> Security Information and Event Management の略。大量のログを集めて、自動で「怪しいパターン」を検知して警告を出すシステム。このデータセットでは **Wazuh**（オープンソースのSIEM）が使われている。

## 第3章：各ファイルの詳細と実際のデータ

### 3-1. Security.evtx — 「誰が・いつ・何をしたか」の基本記録

Windowsセキュリティログ。ログインや権限の使用が中心。

#### 正常マシン（C_Data/30）のイベント分布 — 約4日間・1,477件

| EID | 件数 | 何を意味するか |
|-----|------|-------------|
| 5379 | 454 | 資格情報マネージャーへのアクセス（パスワード管理の利用）|
| **4624** | 297 | **ログイン成功**（誰かがログインした）|
| **4672** | 276 | **特権ログオン**（管理者権限でのログイン）|
| 4799/4798 | 111 | ローカルグループのメンバー確認 |
| **4688** | 45 | **プロセス起動**（何かのプログラムが起動した）|
| 4648 | 22 | 別のアカウントの資格情報を使ってのログイン |

#### 攻撃注入済みマシン（I_Data/1）のイベント分布 — 約1.5時間・8,500件

| EID | 件数 | 何を意味するか | 攻撃との関係 |
|-----|------|-------------|------------|
| **5447** | 5,689 | WindowsFirewall（ファイアウォール）のフィルタ変更 | 攻撃によるFW操作の痕跡 |
| 5156/5158 | 2,119 | ネットワーク接続の許可 | ネットワーク活動の急増 |
| **4702** | 68 | **スケジュールタスクの更新** | **T1053.005（タスクを使った永続化）の直接証跡** |
| 4624/4672 | 50 | ログイン・特権使用 | 攻撃時の認証活動 |

> **正常マシンと攻撃マシンでイベントID分布が全く異なる。**
> 特にEID 4702（タスク更新）は正常マシンにはほぼ出ないが、攻撃マシンでは68件出ており、攻撃の証跡として非常に特徴的。

#### 重要な制約：EID 4688のCommandLine（コマンドライン）が記録されていない

```
実データの例：
  EID: 4688（プロセス起動）
  NewProcessName: C:\Windows\System32\lsass.exe
  CommandLine:    （空）  ← 何のコマンドで起動されたか不明
```

Windowsの標準設定では、プロセス起動時のコマンドラインが記録されない。
**そのためプロセスの詳細追跡はSysmon EID 1を使う必要がある。**

### 3-2. Sysmon.evtx — 研究の主力データ

標準ログでは取れない詳細情報を提供する。このデータセットで最も重要なファイル。

#### 正常マシン（C_Data/30）のイベント分布 — 約4日間・約16,800件

| EID | イベント名 | 件数 | 何を記録するか |
|-----|-----------|------|-------------|
| **1** | ProcessCreate | 275 | プロセス起動。コマンドライン・起動元・ユーザー名・ProcessGuid |
| **3** | NetworkConnect | 3,496 | ネットワーク接続。どのプロセスがどのIPに接続したか |
| 5 | ProcessTerminated | 73 | プロセス終了 |
| 7 | ImageLoad | 1,975 | DLL（プログラム部品）のロード |
| **10** | ProcessAccess | 4,708 | あるプロセスが別のプロセスのメモリにアクセス |
| **11** | FileCreate | 4,145 | ファイル作成。どのプロセスがどのファイルを作ったか |
| **12/13** | RegistryCreate/Set | 4,362 | レジストリの作成・値の設定 |
| **22** | DnsQuery | 153 | DNS問い合わせ。どのドメインにアクセスしようとしたか |
| 26 | FileDeleteDetected | 237 | ファイル削除の検出 |

#### Sysmon EID 1（ProcessCreate）の実データ例

```
時刻:             2024-09-09 21:57:00
Image:            C:\Windows\System32\WindowsPowerShell\v1.0\powershell.exe
CommandLine:      "powershell.exe"
ParentImage:      C:\Windows\explorer.exe         ← 誰が起動したか
User:             DESKTOP-0B7RNLE\win10
ProcessGuid:      {30f6f5d4-f09c-66de-2d01-...}   ← このプロセスの固有ID
ParentProcessGuid:{30f6f5d4-ee99-66de-4a00-...}   ← 起動元プロセスの固有ID
```

> **ProcessGuid（プロセスの一意ID）とは？**
> 各プロセスに割り当てられる「指紋」のようなもの。同名のプロセスが複数起動していても区別できる。
> **ParentProcessGuid（親プロセスの固有ID）** と組み合わせることで、「誰が誰を起動したか」という連鎖を完全に再現できる。

#### プロセス連鎖の復元例（実データから）

```
explorer.exe  → [EID 1: プロセス生成] → powershell.exe
                                              │
                  ┌───────────────────────────┼───────────────────────┐
                  ▼                           ▼                       ▼
      [EID 11: ファイル作成]      [EID 13: レジストリ書込]   [EID 3: NW接続]
      wineventlog\*.csv           HKLM\...\powershell.exe   224.0.0.252:5355
```

この連鎖から「**主体**（powershell.exe / win10ユーザー）が **行為**（ファイル作成・レジストリ書込・ネットワーク接続）を **対象**（各ファイル・レジストリキー・IPアドレス）に対して行い、**連鎖**（explorer.exeから起動された）として説明できる」という復元が可能になる。

これが本研究の「行動復元」で目指していること。

### 3-3. TaskScheduler.evtx — スケジュールタスクの動作記録

スケジュールタスクとは「決まった時間や条件でプログラムを自動実行する仕組み」。
Windowsが内部メンテナンスでも多用しているが、攻撃者が永続化（PCに居座り続ける）ためにも悪用する。

| EID | 意味 |
|-----|------|
| 200 | タスクのアクション開始 |
| 201 | タスクのアクション完了 |
| 100 | タスクエンジン開始 |
| 102 | タスクエンジン停止 |
| 129 | タスクによるプロセス生成 |

正常マシン（C_Data/30、4日間）：約1,100件（Windowsの定期メンテタスクが大半）
攻撃注入済みマシン：攻撃者が作成したタスクが加わり件数増加

### 3-4. Wazuh-Alerts/alerts.json — 自動生成されたアラート

Wazuh（SIEMツール）がEVTXを分析して「これは怪しい」と判断したイベントをJSON形式でまとめたもの。

#### アラート1件分の構造

```json
{
  "_source": {
    "timestamp":  "2024-09-09T15:00:27",
    "agent":      { "name": "VM34" },
    "rule": {
      "id":          "92200",
      "level":       6,
      "description": "Scripting file created under Windows Temp or User folder",
      "mitre": {
        "id":     ["T1059"],
        "tactic": ["Execution"]
      }
    },
    "data": {
      "win": { "eventdata": {
        "image": "C:\\Program Files\\Mozilla Firefox\\firefox.exe"
      }}
    }
  }
}
```

各フィールドの意味：
- `level`：アラートの重要度（0〜15。高いほど深刻）
- `description`：何が検知されたかの説明
- `mitre.id`：対応するMITRE ATT&CKのテクニックID
- `image`：そのイベントを引き起こしたプログラム

#### 重要な観察：正常マシンでも大量にアラートが出る

**C_Data/30（正常マシン・4日間）のWazuhアラート：681件（level 6含む）**

```
具体例：
  Firefox が一時フォルダにファイルを作成
  → Wazuh rule 92200 [level 6] で検知
  → でも実際はFirefoxの正常なキャッシュ保存
  → これが「偽陽性」（本当は正常なのにアラートになること）
```

> **偽陽性（False Positive）とは？**
> 本当は正常・無害な行動なのに、セキュリティツールが「攻撃かもしれない」と誤って警告を出してしまうこと。
> SOCアナリストはこの偽陽性を「これは問題ない」と正しく判断する作業に多くの時間を使っている。

**→ 正常マシン（C_Data）でWazuhがアラートを出すということは、このデータセットには「偽陽性の起点」が豊富に含まれている。これが本研究の実験材料として直接使える。**

### 3-6. GT.yml（I_Dataのみ） — 攻撃の「答え合わせ」ファイル

I_Data（攻撃注入済みマシン）にのみ存在する。「どの攻撃を・どのコマンドで実行したか」が書いてある。

```yaml
attack_technique: T1053.005      # ← MITRE ATT&CKのテクニックID
display_name: 'Scheduled Task/Job: Scheduled Task'
atomic_tests:
  - name: Scheduled Task Startup Script
    executor:
      command: |
        schtasks /create /tn "T1053_005_OnLogon" /sc onlogon /tr "cmd.exe /c calc.exe"
        schtasks /create /tn "T1053_005_OnStartup" /sc onstart /ru system /tr "cmd.exe /c calc.exe"
```

このデータセット全体でカバーされている主な攻撃テクニック：

| MITRE ID | テクニック名 | 攻撃の内容（初心者向け説明）|
|----------|------------|--------------------------|
| **T1053.005** | Scheduled Task | タスクスケジューラーに悪意あるプログラムを登録し、OS起動時などに自動実行させる |
| **T1547.001** | Registry Run Keys | レジストリの「起動時に実行するプログラム」欄に悪意あるファイルを登録する |
| **T1547.004** | Winlogon Helper DLL | Windowsログイン処理に使われるDLLを差し替えて、ログイン時に悪意あるコードを実行させる |
| **T1543.003** | Windows Service | 正規サービスに見せかけた悪意あるWindowsサービスを作成・登録する |
| **T1546.008** | Accessibility Features | アクセシビリティ機能（拡大鏡など）のプログラムを悪意あるものに差し替える |
| **T1574.001** | DLL Search Order Hijacking | Windowsが正規DLLを探す順序を悪用し、悪意あるDLLを先に読み込ませる |
| **T1562.001** | Impair Defenses | セキュリティツール（アンチウイルス等）を無効化する |
| **T1112** | Modify Registry | レジストリを書き換えて設定を改ざんする |

> 論文記載：計 **19テクニック・67サブテクニック** をカバー

#### GT.ymlの重要な制約

GT.ymlには「T1053.005を実行した」という情報はあるが、**「どのEVTXのどのイベントがその攻撃に対応するか」というイベント単位の対応付けはない。**

例えば「schtasks /createを実行した」とGT.ymlに書いてあっても、実際のSysmon.evtxの中でどのEID 1レコードがそれにあたるかは、タイムスタンプを手がかりに自分で突き合わせる必要がある。

## 第4章：データの実態 — 記録期間と規模

### 4-1. 記録期間はマシンによって大きく異なる

このデータセットは「何日間録画した」という統一した録画期間があるわけではなく、**VMのログバッファをある時点でスナップショットしたもの**。そのためマシンごとに期間がバラバラ。

#### C_Data 112台の記録期間分布

| 記録期間 | 台数 | 割合 |
|---------|------|------|
| **1時間未満** | **60台** | **54%（過半数）** |
| 1〜6時間 | 17台 | 15% |
| 6〜24時間 | 5台 | 4% |
| 1〜2日 | 6台 | 5% |
| 2〜4日 | 6台 | 5% |
| **4日超** | **17台** | **15%** |

> 過半数が1時間未満。行動パターンの分析には**4日以上の17台を優先**して使うべき。

長期間（4日超）のマシン候補：

| マシン番号 | 記録期間 |
|-----------|---------|
| C_Data/25 | 約3日2時間 |
| C_Data/27 | 約3日18時間 |
| C_Data/28 | 約3日19時間 |
| **C_Data/30** | **約3日22時間**（最初に実験するのに最適）|

#### I_Data 26台の記録期間

| 区分 | 期間 | 補足 |
|------|------|------|
| 大半（約18台） | 数分〜数時間 | 攻撃実施の時間帯のみ |
| I_Data/4 | 3日4時間 | 比較的長期 |
| **I_Data/15** | **27日17時間** | 長期Persistence確認可能 |
| **I_Data/17** | **9日19時間** | 長期Persistence確認可能 |
| I_Data/22〜24 | 141〜142日 | 古いイベント混入の可能性あり |

> **重要：感染マシスには「攻撃前の正常期間」がほぼない。**
> 「正常だった状態から攻撃が始まった」という変化を同一マシンで追うことはできない。正常の基準はC_Dataと横並びで比較することになる。

### 4-2. VMの環境構成

#### 正常行動の「リアリティ」はGHOSTSが担保

このデータセットの正常マシン（C_Data）では、本物のユーザーが操作したのではなく **GHOSTS（Ghost NPC）** というツールが自動でユーザー行動をシミュレートしている。

GHOSTSは：
- ブラウザで複数のウェブサイトをランダムに閲覧する
- テキストエディタやOfficeでファイルを作成・編集・削除する
- 設定された間隔でこれらの行動を繰り返す

これにより、「何も起きていないまっさらなVM」ではなく、実際のユーザーが使っているPCに近い行動パターンがログに記録される。

## 第5章：研究との接続 — このデータセットで何ができるか

### 5-1. 研究の問い（おさらい）

> **「Hayabusaでアラートが出た正常ログ（偽陽性）を起点として、関連する他のログを探索し、その行動を "主体・行為・対象・連鎖" として再構成（行動復元）できるか？」**

### 5-2. データセットの各要素が研究のどのステップに対応するか

| 研究ステップ | 使うデータ | 具体的な使い方 |
|------------|----------|--------------|
| **① 起点アラートを取得する** | Wazuh-Alerts/alerts.json | C_Data（正常マシン）のアラートを使う。C_Dataは攻撃がないため、出るアラートはすべて偽陽性と確定できる |
| **② 起点アラートの特異性を分類する** | alerts.jsonのrule情報 | アラートが「特定ファイル名（高特異性）」か「IPアドレス（中特異性）」かで分類 |
| **③ 主体（誰が）を特定する** | Sysmon.evtx EID 1 | `Image`（プログラム名）・`User`（ユーザー名）フィールド |
| **④ 行為（何をした）を特定する** | Sysmon.evtx 全EID | EID 11→ファイル作成、EID 3→ネットワーク接続、EID 13→レジストリ変更 |
| **⑤ 対象（何に対して）を特定する** | Sysmon.evtx | `TargetFilename`・`DestinationIp`・`TargetObject` フィールド |
| **⑥ 連鎖（どう繋がっているか）を復元する** | Sysmon.evtx EID 1 | `ProcessGuid` → `ParentProcessGuid` で親子連鎖を追う |
| **⑦ 攻撃との比較をする（第二段階以降）** | I_Data + GT.yml | 同じアラートが攻撃マシンでも出るかを確認、GT.ymlでどの攻撃に対応するかを照合 |

### 5-3. まず試す実験の流れ

```
Step 1: C_Data/30 の5種類のEVTXにHayabusaをかける
         ↓ （C_Dataは正常のみ → 出力はすべて偽陽性と確定）
Step 2: アラート一覧から起点アラートを1つ選ぶ
         例：「Firefox がTempフォルダにファイル作成」
         ↓
Step 3: そのアラートのEVTXイベントを特定し、Sysmon.evtxで関連イベントを探索
         ProcessGuid を使って前後のプロセス連鎖を追跡
         ↓
Step 4: 主体・行為・対象・連鎖を文章で説明できるか評価
         「Firefox（主体）がキャッシュファイルを（行為）Tempフォルダに（対象）保存した。
          Firefox は explorer.exe から起動されており（連鎖）...」
         ↓
Step 5: 復元できた → 成立。できなかった → どこで詰まったかを記録
```

## 第6章：このデータセットの限界と注意点

| 項目 | 内容 |
|------|------|
| **イベント単位のラベルがない** | GT.ymlには「T1053.005を実行した」しか書いておらず、「このSysmonのイベントが攻撃に対応する」という一対一の対応がない。行動復元の正解データとして使うには自分でタイムスタンプを突き合わせる必要がある |
| **Security EID 4688のCommandLineが空** | プロセス起動のコマンドライン追跡はSysmon EID 1を使うこと |
| **過半数のマシンが記録1時間未満** | 長期パターン分析には4日超の17台に絞る |
| **感染マシンに「攻撃前の正常期間」がない** | 同一マシン内で「正常→攻撃への変化」を見ることはできない。C_Dataと横並び比較になる |
| **GT.ymlの書式がマシンごとに統一されていない** | 機械的に読み込む場合は個別の対応が必要 |
| **Description.ymlがないマシンが11台** | C_Data の102〜112番（後から追加されたバッチ）|
| **Ghost NPCの操作とWindowsシステム処理の区別がつかない** | どのログがユーザー模擬操作で、どれがOS自動処理かの区別がない |

## 第7章：データセット選定の根拠（比較表）

| 選定基準 | 重視した理由 | apt-persistence の評価 |
|---------|-----------|----------------------|
| **EVTX形式** | Hayabusaがそのまま使えるため。EVTX以外だとログの情報が欠落してHayabusaのSigmaルールが機能しない | **○** Security + Sysmon が EVTX で配布 |
| **正常データを含む** | 研究の第一段階が「正常行動の復元」のため、攻撃データだけでは実験できない | **○** C_Data 112台（GHOSTSによる正常行動付き）|
| **攻撃/正常の区別** | どちらのログかがわからないとアラートが偽陽性か否かの判断ができない | **△** マシン単位での区別。イベント単位のラベルはない |
| **起点ログから関連ログの追跡** | CLOUSEAUモデルがPOIを起点に関連ログを探索する仕組みのため、追跡可能な構造が必要 | **△〜○** Sysmon ProcessGuidで追跡可能。4688のCommandLineは空 |
| **Hayabusa適合性** | 偽陽性アラートを起点にするフローがHayabusa前提のため必須 | **○** EVTX形式でSecurity+Sysmonの両方に対応 |

**ATLASを不採用とした理由**：ログが txt/json 形式での配布で EVTX 形式でなく、Hayabusa に直接投入できない。アプリケーションログ・DNS ログも Windows イベントログ由来ではなく Sigma ルールが機能しない。
