# ディープリサーチ用プロンプト：研究用データセット探索

---

## 背景・研究概要（リサーチャーへの説明）

以下の研究に使用するデータセットを探している。

**研究テーマ：** SOC Tier2における正常ログを起点とした関連ログ探索・行動復元の自動化  
**研究フェーズ：** 第一段階 — 正常ログを起点として、複数ログソースを横断的に探索し、主体・行為・対象・連鎖を再構成（行動復元）できるかを検証する  
**評価フレーム：** 起点アラートの特異性（高/中/低）× 正常行動の種類（単発操作型/手続型/背景動作型）の9マスで、各組合せごとに探索・復元の成立性を評価する  
**ベースモデル：** CLOUSEAU（POIを起点に関連ログを段階的に探索してストーリー復元するマルチエージェント型LLMシステム）

---

## 現在検討しているデータセットとその欠陥

以下の2つのデータセットを検討したが、いずれも重大な欠陥がある。

### apt-persistence（現在の第一候補）
- **良い点：** 5種類のWindowsネイティブEVTX（Security / Sysmon / System / Application / TaskScheduler）、正常データ112台分（C_Data）、Hayabusa対応
- **欠陥①：** イベント単位のアノテーションがない
  - GT.ymlには「T1053.005を実行した」というテクニックレベルの情報しかなく、「このEVTXのこのイベントIDがこのタイムスタンプで攻撃に関連する」というイベント単位のラベルが存在しない
- **欠陥②：** 関連ログのマーキングがない
  - どのログイベントがどの攻撃チェーンに属するか、正常行動のどのイベント群が1つの行動として関連しているか、というグラフ・連鎖情報がない
  - 行動復元の「正解」を定量評価しにくい

### Python_Evtx_Analyzer / LMD（補完候補）
- **良い点：** 大量サンプル（最大2.3M件）、正常/攻撃のサブセット分離
- **欠陥：** Sysmonのみの単一ログソース。クロスログ探索ができない。イベント単位ラベルなし

---

## 求めるデータセットの評価軸（優先度順）

### 【必須条件】

**1. ログ形式：WindowsネイティブEVTX、またはWindowsイベントログ由来のJSON/XML**
- Hayabusa（WindowsイベントログSigmaルール検知ツール）で直接解析できること
- Sysmon.evtxが含まれること（Event ID 3のネットワーク接続でアプリケーション層の通信を補完できるため）
- Security.evtxが含まれること（認証・プロセス生成イベント）
- 理想：TaskScheduler.evtx / System.evtx / Application.evtx も含む

**2. 正常行動データの存在**
- 攻撃のない純粋な正常ログが含まれること
- 正常ログが現実的なユーザー行動を反映していること（合成ノイズではなく実際の操作ログが望ましい）
- 以下の正常行動種別が含まれていることが望ましい
  - 単発操作型（Web閲覧、ファイル操作、アプリ起動）
  - 手続型（複数操作の連鎖：メール→添付→文書閲覧など）
  - 背景動作型（OSのスケジュールタスク、SearchIndexer、サービス常駐処理）

**3. 複数ログソースが揃っており横断的な探索が可能**
- 最低2種類以上のEVTXソースが同一マシン・同一時間軸で記録されていること
- 異なるログソース間でタイムスタンプが同期されており、イベントをクロス結合できること

---

### 【強く望む条件】

**4. ★イベント単位のアノテーション（最重要の差別化軸）**
- 各ログイベントに対して「攻撃関連 / 正常 / 不明」のラベルがついていること
- 理想：イベントIDとタイムスタンプ単位で攻撃フェーズへの帰属が明記されている
- 例：
  - 「このプロセス生成イベント（Event ID 1, timestamp XX:XX:XX）はT1053.005の実行に直接対応する」
  - 「このネットワーク接続イベント（Event ID 3, timestamp XX:XX:XX）はC2通信に対応する」
- この情報があることで、行動復元の成立/不成立を定量的に評価できる

**5. ★関連ログのマーキング / プロヴェナンスグラフ**
- 1つの行動（攻撃または正常）に関連するログイベント群が明示されていること
- 理想：イベント間の因果関係（プロセスAがプロセスBを生成した、プロセスBがファイルCにアクセスした）がグラフ構造で記録されている
- プロヴェナンスグラフ（来歴グラフ）形式でも可
- この情報があることで「CLOUSEAUが正しく関連ログを特定できたか」の正解データとして使える

**6. MITRE ATT&CK マッピング**
- 攻撃イベントがMITRE ATT&CKのテクニックIDと対応付けられていること
- 特にPersistence（TA0003）、Lateral Movement（TA0008）、Execution（TA0002）のテクニックが含まれることが望ましい

---

### 【あると望ましい条件】

**7. 攻撃と正常が混在した環境のログが含まれること**
- 正常のみのデータ（第一段階）と、攻撃+正常が混在したデータ（第二段階）の両方があること
- 混在データでは、どのイベントが攻撃由来かを判定するためのグランドトゥルースがあること

**8. リアルな環境から収集されたログ、またはそれに近いシミュレーション**
- 実際の企業・組織環境に近い構成（Active Directory、複数ホスト、複数ユーザー）
- 複数のユーザープロファイル（開発者、一般ユーザー、管理者など）が存在することが望ましい

**9. データ量**
- 正常ログが十分に含まれること（最低でも数千イベント、理想は数万〜数十万イベント）
- 複数のシナリオ・複数のマシン分が含まれること

**10. 公開・再現可能**
- 学術研究目的での利用が可能なライセンス
- 論文が存在し、データ収集方法・環境構成が文書化されていること

## 特に期待するデータセットのカテゴリ

以下のカテゴリに属するデータセットを中心に探してほしい：

1. **プロヴェナンスグラフ付きデータセット**
   - DARPA Transparent Computing（TC）プログラムのデータセット（CADETS、THEIA、FIVEDIRECTIONS等）
   - ただし形式がWindowsイベントログ互換かを確認すること

2. **Sysmon中心のWindows攻撃データセット（イベントレベルラベルあり）**
   - OTRFやSecurityDatasets.comが公開しているデータセット群
   - SysmonによるAPT模擬データ

3. **CTF・レッドチーム演習のログデータセット（ラベル付き）**
   - 攻撃手順が明確に記録されており、対応するログが特定されているもの

4. **SOCシミュレーション環境のログ（BRO/Zeek + Sysmon + Windows Event）**
   - 複数ログソースが統合されており、アラートに対応するログチェーンが追跡できるもの

---

## 除外条件

以下に該当するデータセットは除外する：

- Windowsイベントログ（EVTX）由来ではなく、Hayabusaで解析できない形式
  - ネットワークパケットキャプチャ（PCAP）のみのもの
  - NetFlowのみのもの
  - カスタム形式のテキストログのみのもの（CEF/Syslogで構成されるATLAS等）
- 攻撃ログのみで正常行動データがないもの
- イベント単位のラベルも、プロヴェナンスグラフも、GT情報も一切ないもの
- 商用ライセンスで学術利用が制限されるもの

---

## リサーチで答えてほしいこと

各データセットについて以下を調査・報告してほしい：

1. **データセット名・出典論文・公開URL**
2. **ログ形式**（EVTX / JSON / CSV 等）とログ種別（Security / Sysmon / System / Application / TaskScheduler 等）
3. **正常行動データの有無と内容**（何をしているユーザー行動か、件数）
4. **イベント単位のアノテーションの有無**（あればその形式・粒度）
5. **関連ログのマーキング・プロヴェナンスグラフの有無**
6. **MITRE ATT&CKマッピングの有無**
7. **データ量**（イベント数・マシン数・時間範囲）
8. **apt-persistenceと比較した場合の優位点・劣位点**
9. **本研究の9マスフレームワーク（起点アラート特異性 × 正常行動種別）への適合度**

---

## 参考：本研究との関係で特に重要なポイント

- **「正常行動の行動復元」の定量評価ができること**が最重要
  - 復元した「主体・行為・対象・連鎖」が正解のアノテーションと一致しているかを測る必要がある
  - イベント単位のラベルまたはプロヴェナンスグラフがないと、復元結果の正誤が判定できない
- **Hayabusaで偽陽性アラートを抽出できること**
  - 正常行動から発生するHayabusaアラートを起点アラートとして使う
  - そのためには正常ログがWindowsネイティブEVTX形式である必要がある
- **複数ログソースを横断した探索が成立することを検証したい**
  - 1種類のログだけでは横断探索の成立性を検証できない
